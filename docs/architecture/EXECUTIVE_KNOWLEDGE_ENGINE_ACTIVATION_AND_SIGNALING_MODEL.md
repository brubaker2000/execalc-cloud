# Executive Knowledge Engine — Activation and Signaling Model

## Purpose

This document defines how Execalc decides:

- what knowledge to activate
- in what order to activate it
- how strategic signals should surface to the operator

The goal is to ensure that the Executive Knowledge Engine (EKE) behaves with runtime discipline rather than acting as an undifferentiated intelligence pool.

---

# Core Principle

The EKE should activate the **lightest sufficient knowledge set first** and escalate only when the situation requires additional depth, specificity, or current awareness.

This keeps the system:

- fast
- governed
- relevant
- strategically disciplined

---

# Activation Sequence

Knowledge activation should follow this canonical sequence:

1. Scenario detection
2. Domain classification
3. Task type detection
4. Knowledge selection
5. Signal surfacing
6. Strategic synthesis

In shorthand:

Operator input or ambient signal  
→ scenario detection  
→ domain classification  
→ task type detection  
→ knowledge activation  
→ strategic synthesis

---

# 1. Scenario Detection

Scenario detection is the **primary runtime router**.

The system must first determine what type of situation it is dealing with.

Examples:

- negotiation
- due diligence
- decision comparison
- strategic planning
- crisis response
- executive alignment
- monetization analysis
- policy interpretation

Scenario detection determines which knowledge families are even eligible for use.

---

# 2. Domain Classification

Once a scenario is identified, Execalc must determine the relevant domain.

Examples:

- sports
- finance
- healthcare
- real estate
- enterprise software
- manufacturing
- media
- public policy

This narrows the search space and prevents the system from activating irrelevant knowledge assets.

---

# 3. Task Type Detection

The system must also identify what kind of work the operator is asking it to do.

Examples:

- explain
- compare
- recommend
- plan
- diagnose
- summarize
- evaluate risk
- identify monetization paths

Task type determines the reasoning posture and which diagnostics, cartridges, or nuggets are most relevant.

---

# 4. Knowledge Selection Rules

After scenario, domain, and task type are identified, the system should select knowledge according to governance-weighted relevance.

## Always-on baseline

These should always be available:

- Prime Directive
- operator context
- tenant context
- Monolith reasoning substrate
- core governance rules

## Conditional activation

These activate only when relevant:

- thought leadership nuggets
- Execalc runtime cartridges
- client cartridges
- Execalc proprietary data
- client data
- internet search
- free agent contributors

---

# 5. Activation Precedence

Knowledge sources do not carry equal authority.

Suggested precedence order:

1. Security and tenant boundaries
2. Client cartridges / client policy
3. Core governance and Prime Directive
4. Execalc cartridges
5. Monolith
6. Thought leadership nuggets
7. Data lakes
8. Internet search
9. Free agent contributors as temporary supplements

This preserves disciplined reasoning and prevents low-trust or temporary inputs from overriding governed knowledge.

---

# 6. Escalation Logic

The EKE should escalate through knowledge layers only as needed.

## Level 1 — Baseline reasoning
Use:

- Prime Directive
- Monolith
- operator / tenant context

## Level 2 — Structured augmentation
Add:

- nuggets
- Execalc cartridges
- client cartridges
- diagnostics

## Level 3 — Data augmentation
Add:

- Execalc proprietary data
- client data sources

## Level 4 — External supplementation
Add:

- internet search
- free agent contributors

This creates a runtime ladder that balances speed and depth.

---

# Signal Surfacing Model

Execalc must not merely activate knowledge internally. It must also decide what to surface to the operator and how strongly to surface it.

Signals should only be surfaced if they have probable strategic relevance.

---

# Three Signal Levels

## Level 1 — Silent Observation

The system notices a pattern but does not interrupt.

Examples:

- repeated mention of a stakeholder
- recurring theme in Slack
- slow build of urgency around a project
- repeated LinkedIn movement in a category

These signals enrich context but remain background.

---

## Level 2 — Passive Surfacing

The system surfaces the signal in a non-intrusive way.

Examples:

- side panel note
- summary feed item
- daily or session digest
- strategic observation card

This allows the operator to review relevant patterns without unnecessary interruption.

---

## Level 3 — Active Escalation

The system actively surfaces a signal only when it is likely to change what the operator should do.

Examples:

- major opportunity opening
- negotiation leverage event
- policy conflict
- urgent execution bottleneck
- reputational or strategic risk
- monetization signal with asymmetric upside

Active escalation should be rare and justified.

---

# Signal Rule

A signal should surface only if it changes what the operator would rationally do.

This is the core anti-noise rule.

If a signal is interesting but does not alter likely action, it should remain either:

- background context
- passive surfacing
- digest material

---

# UI Surface Mapping

Signals should surface through the interface where they are most relevant.

Examples:

- Chat → recommendations, questions, decision prompts
- Email → opportunity detection, negotiation signals, commitment extraction
- Calendar → meeting prep, context cards, action capture
- Slack → sentiment shifts, blockers, alignment issues
- LinkedIn → market signals, relationship openings, competitive movement
- Admin → governance alerts, connector health, policy conflicts

This keeps signal delivery context-aware.

---

# Operator Posture Modes

Over time, signal intensity should be tunable.

Possible posture modes:

- quiet
- normal
- aggressive
- deal mode
- crisis mode

This allows the same system to support different working styles and situations.

---

# Summary

The Activation and Signaling Model gives the Executive Knowledge Engine runtime discipline.

It ensures that Execalc:

- activates knowledge in a structured order
- respects governance precedence
- escalates only when necessary
- surfaces signals in proportion to their action relevance

This is the bridge between static knowledge architecture and live executive cognition.
