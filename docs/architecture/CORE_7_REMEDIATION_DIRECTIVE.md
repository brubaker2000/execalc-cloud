## Overview

The Core 7 Frameworks are the architectural spine of Execalc. A self-audit of the current build revealed that several frameworks are either underdocumented, missing runtime objects, or have no build-plan presence. This directive names the gap, specifies the required document, and sequences the remediation.

**Chain of authority:** This directive is subordinate to EXECALC_INVARIANTS.md but supersedes any prior gap list in chat history.

## Framework 1 — Prime Directive

### Current Gap
The Prime Directive exists in CLAUDE.md and conceptually in docs/EXECALC_INVARIANTS.md, but there is no runtime enforcement specification. There is no doc that defines how the Prime Directive is applied at the point of inference — which model invokes it, when it fires, what it checks, and what happens when a lens fails evaluation.

### Required Document
`docs/governance/PRIME_DIRECTIVE_RUNTIME_ENFORCEMENT.md`

### Required Content
- Trigger conditions: when does Prime Directive evaluation fire?
- Evaluation sequence: Value/Clarity gate → three-lens evaluation (Assets/Liabilities, Risk/Reward, Supply/Demand)
- Governance failure modes: what constitutes a blind lens vs. an unfavorable lens
- Model tier assignment: Prime Directive evaluation is Judgment-class (Opus-tier)
- Output format: structured evaluation record attached to DecisionArtifact
- Enforcement modes: hard block vs. soft flag vs. advisory
- Runtime object: PrimeDirectiveEvaluation fields (scenario_id, lens_1_status, lens_2_status, lens_3_status, value_or_clarity_output, evaluation_model, evaluation_timestamp, failure_mode_if_any)

### Expected Outcome
Any DecisionArtifact generated by the system can be audited to verify that all three lenses were evaluated. Unevaluated lenses are governance failures, not acceptable gaps.

## Framework 2 — Persistent Memory

### Current Gap
`PERSISTENT_MEMORY_ADMISSION_AND_CLASSIFICATION_DOCTRINE.md` exists and covers the six-test admission filter and classification dimensions well. What is missing is the runtime specification: no MemoryRecord object schema, no promotion ladder implementation spec, no admission endpoint design, and no runtime behavior spec for how admitted memory influences inference.

### Required Document
`docs/architecture/PERSISTENT_MEMORY_SYSTEM.md`

### Required Content
- Memory class taxonomy: Institutional (org-wide), Operator (individual), Session (ephemeral), Canonical (promoted, high-weight)
- Runtime objects:
  - MemoryRecord: memory_id, tenant_id, operator_id, claim_type, claim_text, source_session_id, admission_status, confidence, time_horizon, sensitivity, activation_state, created_at, last_activated_at
  - MemoryPromotion: promotion_id, memory_id, from_state, to_state, promotion_trigger, evaluating_model, timestamp
  - DecisionRecord: references memory_ids that influenced it (traceability)
  - HeuristicRecord: a memory unit that has been promoted to reusable strategic logic
  - OrganizationalSignal: a claim with tenant-wide scope and cross-operator relevance
  - DormantRecord: admitted but not currently active; can be promoted by scenario match
- Promotion ladder implementation: Observed → Candidate → Admitted → Weighted → Canonical
- Admission gate: which model evaluates admission (Judgment-tier), what inputs it receives, what the output record looks like
- Activation mechanics: how admitted memory surfaces during inference (silent vs. passive vs. active)
- Dormancy rules: when does memory go dormant, what re-activates it
- Build slot: Stage 8 (Memory Admission Runtime)

### Expected Outcome
Memory is a governed object, not a side-effect of logging. Every memory unit has a lineage, a state, and a traceable influence path on decisions.

## Framework 3 — Polymorphia (Multi-Actor, Multi-Lens Reasoning)

### Current Gap
Polymorphia is named in CLAUDE.md and referenced in category docs, but it has no standalone doctrine doc and no runtime specification. There is no definition of what a "lens" is as a runtime object, no taxonomy of lens types, and no spec for how multi-lens outputs are synthesized into a coherent response.

### Required Document
`docs/architecture/POLYMORPHIA_RUNTIME_USAGE.md`

### Required Content
- Definition: Polymorphia is the system's ability to hold multiple valid logical perspectives simultaneously without collapsing them prematurely into a single "answer"
- Lens taxonomy: at minimum — Financial Lens, Operational Lens, Risk Lens, Stakeholder Lens, Market/Competitive Lens, Regulatory Lens
- Runtime object: LensActivation (lens_id, lens_type, scenario_id, perspective_output, confidence, conflicts_with_lens_ids)
- Multi-lens synthesis rules: when to surface conflict explicitly vs. resolve it, when to present parallel outputs vs. weighted output
- Governance rule: Polymorphia output must preserve disagreement when disagreement is material; premature synthesis is a governance failure
- Operator posture integration: Workbench mode surfaces all lenses; Control Tower mode surfaces only material divergence
- Model tier: Polymorphia synthesis is Judgment-class (Opus-tier)
- Relationship to Prime Directive: the three lenses of the Prime Directive are a governed subset of Polymorphia lenses

### Expected Outcome
The system can represent organizational reality in multi-dimensional terms — not just "the answer" but the landscape of valid perspectives, with material conflicts named and preserved.

## Framework 4 — Heuristic Coding System

### Current Gap
The Heuristic Coding System has no runtime specification, no schema, and no build-plan slot. It is described philosophically in CLAUDE.md and categorically elsewhere, but there is no doc that defines a Heuristic as a governed object, how heuristics are captured from operator behavior or explicit declaration, how they are encoded, stored, retrieved, and applied.

### Required Document
`docs/architecture/HEURISTIC_CODING_SYSTEM_RUNTIME.md`

### Required Content
- Definition: A heuristic is a reusable, governed strategic logic unit — an operator's pattern of judgment encoded as an activatable asset
- Heuristic schema: heuristic_id, tenant_id, operator_id, heuristic_text, domain, trigger_conditions, confidence, source_type (declared vs. inferred), activation_count, last_activated_at, status (active/dormant/deprecated)
- Capture paths:
  - Declared: operator explicitly states a principle or rule
  - Inferred: system detects consistent decision pattern across sessions and proposes encoding
  - Promoted: a MemoryRecord of type Heuristic is promoted to HeuristicRecord
- Invocation paths:
  - Scenario-triggered: heuristic matches scenario domain and trigger conditions
  - Operator-invoked: operator explicitly calls a named heuristic
  - Reflex-triggered: a Reflex fires and activates associated heuristics
- Governance rules: heuristics must be attributable (no anonymous heuristics), reviewable, and revocable
- Network promotion: heuristics may be promoted from operator-scope to tenant-scope to Execalc-global scope (with appropriate governance gates at each level)
- Build slot: Stage 8+ after Memory Admission Runtime is stable

### Expected Outcome
Operator judgment is not lost between sessions. Patterns of good reasoning become durable assets that compound over time. The system gets smarter about this operator — not just smarter in general.

## Framework 5 — Recursive Reintegration

### Current Gap
Recursive Reintegration is the weakest-documented framework in the repo. It exists in name only. There is no doc, no runtime object, no design specification, and no build-plan slot. The concept — that past reasoning should feed future reasoning, that the system should examine its own outputs against doctrine — has no implementation pathway.

### Required Document
`docs/architecture/RECURSIVE_REINTEGRATION_MODEL.md`

### Required Content
- Definition: Recursive Reintegration is the system's capacity to treat its own prior reasoning as input to current reasoning — not replay, but governed reuse
- What "recursive" means here: the system applies its own analytical procedures to its own outputs to check for drift, incompleteness, or doctrine violation
- Three recursive loops:
  1. Session loop: within a session, prior outputs are available as governed context
  2. Cross-session loop: admitted memories and prior DecisionArtifacts are retrievable and can be explicitly invoked
  3. Doctrine loop: the system can evaluate any output against GAQP principles and Prime Directive governance on request or by reflex
- Runtime objects:
  - ReintegrationTrigger: trigger_id, scenario_id, trigger_type (manual/reflex/scheduled), target_artifact_id, evaluation_criteria
  - ReintegrationOutput: output_id, trigger_id, prior_artifact_id, evaluation_result, drift_flags, doctrine_compliance_score, recommendations
- Minimum initial capability: "given this current scenario and this prior DecisionArtifact on a related scenario, identify material divergence and unresolved contradictions"
- Governance rule: Recursive Reintegration does not override prior decisions — it surfaces them for operator judgment
- Model tier: Judgment-class (Opus-tier) for doctrine evaluation; Labor-class acceptable for retrieval and normalization
- Build slot: Stage 9+, after Memory and Heuristic systems are stable

### Expected Outcome
The system accumulates institutional intelligence. Decisions do not begin from zero. Prior reasoning is available, auditable, and materially influences current synthesis. The operator builds compound judgment advantage over time.

## Framework 6 — Proactive Solutions Architecture

### Current Gap
PSA has no standalone doc, no runtime specification, no reflex schema, and no trigger taxonomy. It is mentioned in CLAUDE.md under Core 7 and described as "system initiates before asked," but the mechanism — what detects a scenario, what constitutes a proactive trigger, what the output looks like, how it differs from a standard response — is not defined anywhere.

### Required Document
`docs/architecture/PROACTIVE_SOLUTIONS_ARCHITECTURE.md`

### Required Content
- Definition: PSA is the system's capacity to initiate analysis, surface signals, and propose actions before the operator explicitly requests them — based on scenario detection, pattern recognition, and pre-loaded response templates
- Distinction from reactive response: PSA fires on detection, not on query; output is offered, not requested
- Reflex taxonomy (minimum):
  - Risk Reflex: detects risk indicators in operator input or ambient signals and surfaces without being asked
  - Opportunity Reflex: detects structural advantage signals and proposes them
  - Contradiction Reflex: detects conflict between current input and prior admitted memory
  - Urgency Reflex: detects time-sensitivity and escalates posture
  - Governance Reflex: detects doctrine violation candidate and flags it
- Runtime objects:
  - ReflexDefinition: reflex_id, reflex_name, reflex_type, trigger_pattern, activation_threshold, output_template, governing_framework, model_tier, status
  - ReflexActivation: activation_id, reflex_id, scenario_id, trigger_evidence, output_generated, operator_acknowledged, timestamp
- Pre-loaded response templates: structured outputs that can be populated rapidly when a reflex fires; not free-form generation
- Governance rules: PSA cannot execute actions autonomously; it surfaces and offers; operator approval required for action
- Operator posture integration: Control Tower mode maximizes PSA surface area; Workbench mode suppresses PSA unless explicitly invoked
- Build slot: Stage 9+ after EKE activation is stable

### Expected Outcome
The system behaves as a governed early-warning and opportunity-detection layer. The operator is not solely responsible for knowing what questions to ask. The system watches, detects, and offers — within governed boundaries.

## Framework 7 — Executive Knowledge Engine (EKE Admission and Activation Governance)

### Current Gap
The EKE has strong stratum documentation (EXECUTIVE_KNOWLEDGE_ENGINE_STRATA.md) and a solid activation signaling model (EXECUTIVE_KNOWLEDGE_ENGINE_ACTIVATION_AND_SIGNALING_MODEL.md). What is missing is an explicit governance policy: who admits knowledge assets into each stratum, what the admission criteria are per stratum, how trust weighting is assigned, and what the activation precedence rules are in cases of conflict between strata.

### Required Document
`docs/governance/EKE_ADMISSION_AND_ACTIVATION_POLICY.md`

### Required Content
- Admission criteria per stratum:
  - Monolith: peer-reviewed, multi-source corroboration required; Execalc team governed
  - Thought Leadership Nuggets: six-test filter required; source attribution mandatory
  - Execalc Runtime Cartridges: validated against invariants before admission; versioned
  - Client Cartridges: tenant-governed; must not conflict with Security Layer
  - Execalc Proprietary Data Lake: structured ingestion only; provenance required
  - Client Data Lake: API-governed; tenant authorization required
  - Internet Search: lowest trust; supplementary only; never sole source for a governed claim
  - Free Agent Contributors (FAR-01): session-bound; no persistence without explicit admission
- Trust weighting schema: trust_score per stratum + source_corroboration_count + recency_weight
- Activation precedence (conflict resolution):
  1. Security Layer (absolute veto)
  2. Client Cartridges (tenant policy)
  3. Prime Directive governance
  4. Execalc Runtime Cartridges
  5. Monolith
  6. Thought Leadership Nuggets
  7. Proprietary Data Lake
  8. Client Data Lake
  9. Internet Search
  10. Free Agents
- Conflict resolution rule: higher-precedence stratum wins; conflict must be logged, not silently resolved
- Governance failure mode: activating a lower-trust stratum when a higher-trust stratum has a conflicting asset is a governance violation
- Model tier: Admission evaluation is Judgment-class; retrieval is Labor-class
- Versioning: all cartridges and Monolith content is versioned; rollback must be possible
- Build slot: Stage 8B.8 (Memory Runtime Scaffolding) → Stage 9 (EKE Activation)

### Expected Outcome
The EKE is not a retrieval system — it is a governed knowledge activation system. Every piece of knowledge has a trust tier, a source, and a precedence position. Conflicts are visible. Admissions are auditable.
