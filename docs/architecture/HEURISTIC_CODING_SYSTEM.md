## Canonical Definition

Heuristic Coding System is Execalc's framework for converting strategic instinct into modular, tagged, and governable logic units that can be stored, audited, recalled, and deployed across future decisions instead of evaporating as thought soup.

Each encoded heuristic is tagged by source, domain, and confidence; cleared through the Prime Directive; governed by activation protocols in MDL; stored in Persistent Memory; audited by Recursive Analysis; and callable by Proactive Solutions Architecture for preemptive action.

---

## Why This Exists

Uncoded instinct is not durable intelligence.

When an operator expresses a flash of strategic insight in a session — a pattern they have learned from a decade of negotiations, a rule of thumb that has saved them in capital raises, a read on human behavior that consistently predicts how people behave under pressure — that insight passes through the conversation and evaporates. The next session begins without it. The session after that begins without it. The system accumulates transcripts but not judgment.

Without Heuristic Coding, strategic insight remains soup: it floats in chat context, never connects to other ideas, never becomes activatable, and never compounds into system intelligence. The operator has to re-express the same insight every time they want the system to use it.

Heuristic Coding exists to stop that evaporation. Not by remembering better — by converting the ephemeral into the governed.

> The framework is not note-taking. It is the conversion of intuition into executable structure.

## The Canonical Illustration: The Willie Mays Problem

Willie Mays, in his prime, caught fly balls in ways that no coaching manual fully explained. He read the ball off the bat before most players had processed the crack of contact. That read was not random — it was a compressed, experience-derived pattern: bat angle, pitch location, batter's swing mechanics, wind. He had encoded years of observation into a reflex that operated faster than conscious analysis.

The question Heuristic Coding answers is not: "Can this instinct be remembered?"

It is: **"Must this instinct be encoded, tagged, and made callable later?"**

An operator who says "I've noticed that when a buyer delays signing after pricing has been agreed, it almost always means there's an internal approval issue they haven't disclosed — not a budget problem" has just expressed a Willie Mays catch. Left in the conversation, it is a good observation. Encoded as a heuristic, it becomes:

- A logic unit that activates whenever the system detects buyer delay signals in a negotiation scenario
- A governed claim with source attribution, confidence level, and activation scope
- An asset that PSA can deploy proactively in the next negotiation session, before the operator thinks to ask
- An entry that Recursive Analysis will validate or challenge over time as Decision Outcomes accumulate

The difference between admiring the catch and encoding the reflex is the entire product.

## The Five Core Functions

### 1. Interception

Capture operator insight that would otherwise pass as casual language.

The system must recognize when an operator has expressed something worth encoding — a pattern, a rule of thumb, a causal claim derived from experience — and flag it for encoding rather than letting it scroll by as conversational context.

Interception is not passive recall. It is active signal recognition. The interception reflex (BDR-01, see below) is what makes this proactive rather than dependent on the operator explicitly saying "encode this."

**What interception looks for:**
- Conditional structure: "when X happens, Y follows" or "if X, then Y"
- Derived patterns: "I've noticed that," "in my experience," "what I've learned is"
- Compressed rules: "always," "never," "the key is," "the real issue is"
- Attribution signals: insight traced to a specific source, situation, or track record

---

### 2. Encoding

Translate the intercepted insight into a modular logic unit rather than a one-off memory.

A one-off memory stores what was said. A logic unit stores what can be used. The difference is structure: the encoded heuristic has a defined claim, a defined scope, defined activation conditions, and defined metadata. It is not a quote from a session — it is a governed claim that happens to have originated in a session.

Encoding uses the GAQP 19-type taxonomy. Most operator-derived heuristics encode as one of:
- **Heuristic** — a single-move decision rule derived from experience
- **Causal Claim** — an operator-observed mechanism (X drives Y in this context)
- **Tendency** — a probabilistic pattern the operator has observed repeatedly
- **Diagnostic Signal** — an indicator the operator has learned to read
- **Stratagem** — a multi-move sequence the operator has found effective

The encoding step produces the structured object. The remaining functions govern it.

---

### 3. Tagging

Assign governance metadata so the logic unit is manageable.

Every encoded heuristic receives the full GAQP governed claim metadata schema:

| Field | What it captures |
|---|---|
| `claim_type` | From the 19-type taxonomy |
| `source` | Operator-derived; named source if applicable |
| `domain` | e.g., negotiation, capital allocation, team dynamics |
| `confidence_level` | Initial confidence set at encoding; updated by Recursive Analysis over time |
| `activation_scope` | Universal / domain / situational / tenant-specific |
| `polarity` | Positive / cautionary / negative / neutral / mixed |
| `durability_class` | Enduring / medium-term / ephemeral |
| `freshness_rule` | Timeless / date-sensitive / event-bound / expiring |
| `activation_tags` | Situation Class, Dynamic Class, PD Lens (from GAQP_EKE_ACTIVATION_BRIDGE.md) |

Tagging is what makes the heuristic governable. An untagged instinct cannot be audited, cannot be selectively activated, and cannot be retired when conditions change. Tagging is not bureaucracy — it is the difference between a logic asset and an undifferentiated note.

---

### 4. Storage and Recall

Place the encoded and tagged heuristic in the Heuristic Library within Persistent Memory, where it is available for later retrieval and matchmaking.

The Heuristic Library is a governed subset of Persistent Memory. It holds:
- Operator-derived heuristics (captured from sessions)
- Promoted network heuristics (elevated from the EKE thought leadership corpus when an operator validates them in their specific context)
- Organization-specific causal claims and tendencies

Recall operates through the activation tag intersection model (see `GAQP_EKE_ACTIVATION_BRIDGE.md`). A heuristic in the library fires when its activation tags match the current situation's tags above the relevance threshold. Recall is not keyword search. It is situational pattern matching against the operator's pre-encoded judgment.

---

### 5. Mesh Behavior

Allow one coded heuristic to combine with others to trigger overlays, scenarios, or future action.

A single heuristic is useful. A cluster of heuristics that reinforce each other, create an activation pattern, or jointly constitute a strategic posture is more than the sum of its parts.

Mesh behavior occurs when:
- Multiple heuristics with overlapping activation tags all fire in the same session — the system detects the cluster and may elevate to Carat consideration
- A heuristic pairs with an EKE corpus entry to produce a combined activation signal stronger than either alone
- A set of encoded causal claims form a coherent mechanism that can be named and governed as a Mental Model
- An operator's coded heuristics, combined with observed tendencies and Decision Outcomes, constitute an emerging organizational pattern worth surfacing to PSA

Mesh behavior is why Heuristic Coding compounds. Individual heuristics are logic atoms. Their combinations are emergent strategic intelligence.

## BDR-01: The Interception Reflex

BDR-01 (the Blind Starving Man reflex) is the pre-loaded pattern recognition reflex that triggers heuristic creation. It is the mechanism that makes interception proactive rather than dependent on the operator flagging their own insight.

**The Blind Starving Man doctrine:** Ordinary AI does not reliably perceive leverage, implication, or operator brilliance because it has not been told to look for those things. It processes what it is asked. A person standing in a kitchen, blind and starving, does not benefit from the food that surrounds them. The awareness must be explicitly coded.

BDR-01 is that coded awareness. It fires when the system detects linguistic or structural signals indicating that an operator has expressed something worth encoding — and it triggers the Heuristic Coding encoding pipeline without waiting for an explicit instruction.

Without BDR-01: the operator must explicitly say "encode this as a heuristic." Most operators will never do that. The insight evaporates.

With BDR-01: the system intercepts the signal, proposes the encoding, and awaits operator confirmation before admission. The operator's burden drops to approval, not initiation.

**The developer implication:** Any implementation of BDR-01 must include auto-tagging logic, Heuristic Library storage, and recall/matchmaking integration. BDR-01 without the coding pipeline behind it is a notetaker. With the pipeline, it is an architect of judgment.

## The Prime Directive Admission Gate

Heuristic Coding does not make instincts sovereign. They must clear governance before entering the library.

Every encoded heuristic is evaluated against the Prime Directive before admission:

1. **Value or clarity test:** Does this heuristic, if activated, guide the operator toward value (monetization) or clarity (operational decision)? A heuristic that leads toward neither fails this test.

2. **Three-lens evaluation:** Does the encoded claim have a position on assets/liabilities, risk/reward, and supply/demand? An encoded causal claim that is indifferent to all three PD lenses is either too vague to be useful or is not actually a strategic heuristic.

3. **Non-harm check:** Does activation of this heuristic in governed contexts risk producing recommendations that violate tenant constraints, compliance requirements, or ethical limits?

Heuristics that clear all three are admitted to the Heuristic Library. Heuristics that fail are returned to the operator with a note explaining why the instinct, as expressed, did not pass governance. The operator may reframe and resubmit.

This gate is what separates Heuristic Coding from intuition worship. The system does not merely preserve what the operator believes. It preserves operator brilliance inside a value-and-risk filter.

## Relationship to the GAQP Artifact Standard

Heuristic Coding System is the operational runtime expression of GAQP extraction applied to operator-generated knowledge.

The GAQP Artifact Recognition and Extraction Standard defines how to identify and type artifacts from text. Heuristic Coding System is what happens when that extraction runs on live operator input in a session rather than on a book or document.

The pipeline is identical:
```
Extraction (interception) → Classification (encoding + typing) → Admission (PD gate + seven-test filter)
```

The difference: external document extraction populates the EKE corpus with received wisdom from the thought leadership bench. Session-based heuristic coding populates the Heuristic Library with operator-derived wisdom. Same standards. Same governance. Different sources.

This is also why the 19-type taxonomy matters at runtime, not just as a classification exercise. When BDR-01 intercepts an operator insight, the encoding step must assign one of the 19 types — not to satisfy a taxonomy, but because the type determines how the heuristic will activate, how it will be stored, and how Recursive Analysis will audit it.

## Relationship to the EKE Corpus

The EKE corpus (Framework 4) holds the thought leadership bench: Sun Tzu, Voss, Munger, Drucker, and the 230+ seed entries representing institutional wisdom.

The Heuristic Library (Framework 5) holds the operator bench: the strategic instincts, causal claims, and decision rules that this specific operator has developed through experience.

They are separate stores with the same governance standard. At activation time, both are queried. A session may activate a Voss negotiation heuristic from the EKE corpus AND an operator-coded buyer delay heuristic from the Heuristic Library simultaneously. The activation bridge (see `GAQP_EKE_ACTIVATION_BRIDGE.md`) handles both using the same tag-intersection model.

The compounding structure: operators who use Execalc over time build a Heuristic Library that is calibrated to their specific industry, counterparties, and organizational context. The EKE corpus provides universal received wisdom. The Heuristic Library provides contextually validated wisdom. Together they produce synthesis that neither could produce alone.

## Developer Handoff Requirements

Heuristic Coding System must never be treated as a vague aspiration to "remember better." It implies concrete runtime objects and behaviors:

**Required runtime objects:**
- Heuristic admission endpoint — receives encoded candidate, runs PD gate and seven-test filter
- Heuristic Library store — governed subset of Persistent Memory with activation tag index
- Auto-tagging component — assigns GAQP metadata and activation tags at encoding time
- Recall/matchmaking engine — queries the Library using tag-intersection scoring from the activation bridge
- BDR-01 interception signal — fires when operator input contains heuristic-signal linguistic patterns

**Required behaviors:**
- Interception fires proactively — operator does not need to explicitly request encoding
- Every admitted heuristic carries full GAQP metadata schema
- Heuristic Library entries are linked to the Decision Outcomes that reference them (for Recursive Analysis audit)
- Mesh detection — system flags when multiple heuristics with overlapping activation tags fire simultaneously
- Lifecycle management — heuristics transition through states (Active → Flagged → Revalidated/Retired) based on Recursive Analysis findings
