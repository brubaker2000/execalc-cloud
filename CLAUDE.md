## 1. What Execalc Is

### Founding Sentence
> Execalc began as the idea that the astonishing utility of a chatbot could be transformed into something vastly more powerful if it were governed, disciplined, and aligned with the operator's interests.

### Category Statement
Execalc is a **governed, normative, and informed** cognitive operating system for executive judgment.

It is not a chatbot. It is not an agent framework. It is not a dashboard.

**Governed** — the regulative layer: enforces safety, compliance, and tenant controls. Every responsible AI system has this. It is necessary but not the product.

**Normative** — the constitutive layer: the Prime Directive, the three lenses, the Core 7 frameworks. These are not restrictions on the model. They are what the system is. The system is built to pursue value and clarity, not merely prevented from harm. This is what separates Execalc from a guardrailed chatbot.

**Informed** — the knowledge layer: the EKE corpus, the thought leadership bench, the heuristic library, organizational memory. These activate on the basis of situation recognition, not keyword search.

The LLM is the engine. The normative architecture and informed corpus are the product.

**Category thesis:** Execalc is governed, normative, and informed intelligence infrastructure — the system that converts machine prediction into disciplined organizational judgment.

### Organizing Principle
Execalc must compress the organizational decision cycle until the only remaining bottleneck is human judgment.

### Workflow Thesis
Legacy workflow: `email → document → meeting → spreadsheet → deck → meeting → decision`

Target workflow: `signal → synthesis → decision artifact → action → memory`

The long-term advantage of governed AI is not speed alone — it is the ability to perceive patterns previously invisible to human cognition.

### Technology Compression Thesis
Each era compressed time between information and action:
- Stagecoach → Telegraph → Telephone → OS → Excel → LLM → **Execalc**

Excel compressed time-to-quantitative-analysis.  
LLMs compress time-to-qualitative-synthesis.  
Execalc is different: it does not merely move information faster — it **pre-encodes judgment**.

The grandmaster analogy: grandmasters don't calculate faster in speed chess — they pattern-match to pre-loaded frameworks. Execalc eliminates the agonizing so the operator can focus on the irreducible.

### Product Iterations (One Governed Substrate, Four Surfaces)
1. Governed chatbot / decision aide
2. Corporate namespace synthesis
3. Intelligent Front Door (IFD)
4. Bridge between organizational namespaces

IFD and Bridge are future build scope, not current.

## 2. GAQP — Generally Accepted Qualitative Principles

**GAQP is the doctrine. Execalc is the operating system.**

GAQP is to qualitative judgment what GAAP is to financial accounting.  
GAAP disciplined money. GAQP disciplines meaning.

McKinsey and BCG do proto-GAQP privately and manually.  
GAQP is the explicit, portable, platform-native standard they do not have.  
Execalc is not consulting in software form.

### GAQP v0.1 Core Principles (10)
1. **Materiality** — Only signal that changes what a decision-maker would rationally do is worth capturing.
2. **Traceability** — Every claim must trace back to its source, context, and admission path.
3. **Authenticity** — Claims must reflect what was actually observed, not what was hoped.
4. **Context** — No claim is fully interpretable without its originating context.
5. **Separation of Observation and Interpretation** — Raw observation and derived interpretation must be stored and labeled separately.
6. **Corroboration** — Claims gain weight through independent confirmation; single-source claims carry lower confidence.
7. **Explicit Assumptions** — Assumptions embedded in claims must be named, not hidden.
8. **Revision** — Claims must support structured revision with lineage preserved; revision is not deletion.
9. **Governed Memory Admission** — Not all claims are admitted to active memory; admission requires a governed evaluation.
10. **Reconstructable Decision Lineage** — Any decision output must be reconstructable from its admitted claims, applied frameworks, and governing logic.

These 10 principles are v0.1 and subject to deliberate versioned revision. Do not add principles casually.

## 3. The Atomic Nugget (Governed Claim)

**GAQP does not govern raw language; it governs claims extracted from language.**

The atomic nugget is the smallest durable unit of qualitative reasoning. It is not a sentence. It is a **governed claim** — a unit that has passed admission criteria and been classified.

### Six-Test Admission Filter
A claim must pass all six to be admitted:
1. **Stand-alone** — Can it be understood without its surrounding conversation?
2. **Disputability** — Can it be challenged? Pure truism fails this test.
3. **Governance** — Does it fall within a recognized claim type taxonomy?
4. **Activation** — Is there a scenario in which this claim would change a decision?
5. **Durability** — Is it likely to remain relevant beyond its originating conversation?
6. **Composability** — Can it combine with other claims to form higher-order reasoning?

### Claim Type Taxonomy (v0.1)
- Axiom, Definition, Principle, Heuristic, Best Practice, Tendency
- Observation, Event, Constraint, Objective
- Tradeoff, Causal Claim, Diagnostic Signal

### Three-Layer Object Structure
1. **Object class** — Governed Claim
2. **Claim type** — from taxonomy above
3. **Governance metadata** — source, confidence, scope, activation state, time horizon, sensitivity, provenance

## 4. Prime Directive — Canonical Model

The Prime Directive is the top-level governance framework. It determines whether something gets acted on.

### Structure (two-tier)

**Tier 1 — The Objective:**  
A decision or action must deliver **value** (monetization strategy) or **clarity** (operational clarity).  
Clarity is itself a form of value. These are the only two valid outputs the Prime Directive recognizes.

**Tier 2 — The Three Lenses:**  
Whether value or clarity is delivered is judged through three lenses:

1. **Assets vs. Liabilities** — What does this do to the balance sheet position?
2. **Risk/Reward** — Is the risk/reward balance favorable and explicit?
3. **Supply/Demand** — Is there a structural imbalance being leveraged or exposed?

Anything that does not pass these gates does not get acted on.

### Enforcement Doctrine
**The Prime Directive does not require all three lenses to be positive.**  
It requires all three to be **evaluated**.

> "The sin isn't imbalance. The sin is blindness."

A decision where any lens was never evaluated is a governance failure.  
A decision with one unfavorable lens is valid — if that lens was seen, named, and the tradeoff was accepted knowingly.

## 7. Multi-Model Routing (Locked Invariant)

**Model selection is governed by trust tier, not cost preference.**

| Layer | Role | Model Tier |
|---|---|---|
| Prime Directive evaluation | Judgment | Premium (Opus-class) |
| Memory admission gate | Judgment | Premium |
| Reflex gate decisions | Judgment | Premium |
| Summarization | Labor | Cheapest safe |
| Normalization | Labor | Cheapest safe |
| Extraction | Labor | Cheapest safe |

This is a build invariant. Cost optimization that downgrades trust-critical layers violates the invariant. The right question for any new capability: "Is this trust-critical or labor?" — that determines model tier.

## 8. Memory Doctrine

### Core Doctrine Line
> "Journaling is not memory."

Storing conversation turns is not memory. Memory requires governed admission.

### Admission Criteria
Every candidate memory unit must satisfy:
- **Materiality** — Does it change what the operator would rationally do?
- **Durability** — Will it remain relevant beyond this session?
- **Provenance** — Is the source known and credible?
- **Decision Relevance** — Could it activate in a future governed decision?

### Memory Promotion Ladder
```
Observed → Candidate → Admitted → Weighted → Canonical
```
- **Observed** — noticed in a session, not yet evaluated
- **Candidate** — flagged for admission review
- **Admitted** — passed six-test filter; in active or dormant memory
- **Weighted** — confidence and relevance scored and updated
- **Canonical** — durable, high-confidence, broadly activating

### Memory Classification Dimensions
family | source kind | activation state | scope | confidence | time horizon | sensitivity

### Active vs. Dormant
- **Active** — eligible to influence runtime reasoning
- **Dormant** — stored and recallable; does not trigger reflexes until promoted

## 9. Signal Elevation Doctrine

Bolding is a governed extraction system, not a formatting habit.

Rail-candidacy test for a signal: **durable, distinct, structurally important, able to stand outside the conversation flow.**

Elevation ladder:
```
Bold-worthy → Rail-worthy → Memory-worthy → Doctrine-worthy
```

A signal surfaced in chat is not memory. It is a candidate.  
Only governed admission makes it memory.

### Push Protocol
```bash
git push -u origin claude/codebase-audit-ltOXr
```

### Commit Style
Descriptive present-tense subject line, 72 chars max.  
Reference stage or doc set being advanced.

### CI Gate
Before any push verify:
```bash
python -m compileall -q src/service
pytest -q
```
All tests must pass. DB integration tests may show as errors when local Postgres is absent — this is expected.
